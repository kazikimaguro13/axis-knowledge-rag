# RAGアーキテクチャの設計判断

RAG (Retrieval-Augmented Generation) は、外部知識をベクトル検索で取り出し、LLM の入力に差し込むことで「学習時点に存在しない情報」や「ローカル固有のナレッジ」に答えられるようにするパターンである。素朴に組むと「ベクトル検索→上位 N 件を全部詰め込んで生成」となるが、本番運用では次の論点が必ず出てくる。

第一に、検索の粒度。ドキュメント単位で投げ込むと文脈は豊富だが、ノイズで関連度が薄まり、生成側が「結局どこを参照すべきか」を判断しきれない。チャンク単位にすると関連度は鋭くなる代わりに、前後文脈を失って誤読しやすい。実務的には、章 / 節単位で軽く区切り、ヘッダ階層をメタデータとして残しておく折衷案が安定する。

第二に、検索結果の合成戦略。素朴な concat ではコンテキストウィンドウをすぐ食い潰すので、上位スコアを優先しつつ、軸メタデータ (category, topic, level) によるリランクや、重複排除を入れるとよい。本プロジェクトでは「軸フィルタ + ベクトル類似度のハイブリッド」をコアに据える。
