# プロンプトエンジニアリングの実務原則

プロンプトエンジニアリングは「LLM への入力をどう設計すれば、安定して期待出力を引き出せるか」を扱う実務領域だ。研究的な技法 (Chain-of-Thought, ReAct, Tree-of-Thoughts など) は重要だが、業務システムで効くのは地味な原則の積み重ねである。

第一に、役割と制約を冒頭で固定する。「あなたは X です」「以下の制約に従ってください」という明示的な指示は、後段の指示ブレや出力ゆれを大きく減らす。第二に、出力フォーマットは例示する。JSON Schema を口頭で説明するより、期待形式の具体例を 1〜2 件貼った方が遵守率が高い。第三に、否定形より肯定形で書く。「〜してはいけない」は守られにくく、「常に〜する」のほうが安定する。

RAG 文脈では、検索した出典をプロンプトに差し込む際の「区切り方」が品質に直結する。`<source id="doc_001">...</source>` のような構造化タグで囲い、生成側に「出典 ID を必ず引用する」と指示すると、ハルシネーション率がはっきり下がる。後段で出典リンクを UI に表示する際にも、この ID が手がかりになる。

なお、本ドキュメントの `refs` には敢えて存在しない `doc_999` を含めてある。Week 2 で実装する参照整合性チェックの動作検証用サンプルだ。
