# RAG の評価指標と運用観点

RAG システムの品質を「肌感」で語っているうちは、リグレッションが起きても気づけない。最低限の評価軸を持っておくと、プロンプト変更や埋め込みモデル差し替えの判断が定量化できる。

評価は 2 段階に分けると整理しやすい。第一段階が「Retrieval の質」で、Recall@K (正解ドキュメントが上位 K 件に入る割合) と MRR (正解の順位の逆数平均) を見る。doc_001 で論じた検索粒度の選択 (章単位かチャンク単位か) は、ここの数字で判定できる。第二段階が「Generation の質」で、出典忠実度 (faithfulness)、回答妥当性 (answer relevance)、ハルシネーション率を見る。Ragas や TruLens のような既製ライブラリもあるが、まずは 30〜50 件の手作り評価セットで十分始められる。

運用に乗せると見えてくるのが、コスト・遅延・品質のトレードオフ。Top K を増やせば再現率は上がるがコンテキスト消費が増え、生成が遅く高くなる。本プロジェクトでは Top K を UI から可変にして、評価結果と突き合わせながら既定値を調整する方針だ。
